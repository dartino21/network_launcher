# Решение проблем

Начинайте с вкладки **Логи** и файла `data/logs/app.log` рядом с приложением. Сообщения идут по этапам: распознавание проекта → proxy → ngrok → локальный сервер → локальная проверка → публичная проверка.

## Программа не запускается

### EXE открыт прямо из ZIP

Полностью распакуйте архив в папку, доступную для записи, и запустите EXE оттуда. Приложение создаёт рядом `data/` и локальный журнал.

### Windows показывает предупреждение

Текущая сборка не подписана коммерческим сертификатом. Загружайте ZIP только из [Releases](https://github.com/dartino21/network_launcher/releases) этого репозитория. Если политика вашего компьютера запрещает неподписанные приложения, запустите проект из исходников или обратитесь к администратору.

### Запуск из исходников сразу закрывается

Откройте PowerShell в корне репозитория и выполните:

```powershell
python -m pip install -r requirements.txt pytest
.\scripts\run_dev.ps1
```

## Папка проекта не распознаётся

В корне должен находиться хотя бы один маркер:

- Compose: `docker-compose.yml`, `docker-compose.yaml`, `compose.yml` или `compose.yaml`;
- static: `index.html`;
- Node.js: `package.json`;
- Flask: `app.py`.

Выбирайте корень проекта, а не `src/`, `dist/`, отдельный HTML-файл или исполняемый файл.

## ngrok не настроен или ссылка не появляется

1. Получите токен в [ngrok Dashboard](https://dashboard.ngrok.com/get-started/your-authtoken).
2. Сохраните его на вкладке **Настройки** и дождитесь зелёного индикатора.
3. Проверьте доступ в интернет.
4. Убедитесь, что другой ngrok не занял локальный Agent API на порту `4040`.
5. При `ERR_NGROK_9040` смените сеть или используйте разрешённый вашей политикой VPN.

Альтернатива для разработчика:

```powershell
$env:NGROK_AUTHTOKEN = "<token>"
.\scripts\run_dev.ps1
```

Не публикуйте реальный токен в issue, скриншоте или логе.

## Node.js-проект не запускается

- Установите актуальную LTS-версию Node.js и перезапустите Network Launcher.
- Проверьте `node --version` и `npm --version` в новом PowerShell.
- Убедитесь, что в `package.json` есть рабочий script `start`.
- Если существует `build/`, приложение использует `npx serve -s build`; для dev/HMR удалите устаревший build или запускайте ожидаемый режим вручную.
- Посмотрите объединённый stdout/stderr на вкладке **Логи**.

Установка зависимостей может занять время при первом запуске. Не закрывайте приложение, пока журнал показывает активную установку.

## Flask-проект не запускается

Network Launcher ищет Python в `.venv`, `venv`, `env`, затем в системном PATH. Рекомендуемая подготовка внутри проекта:

```powershell
py -3.12 -m venv .venv
.\.venv\Scripts\python.exe -m pip install flask
```

`app.py` должен запускать сервер на порту из окружения/настройки проекта. Если код всегда использует фиксированный другой порт, укажите его как **Frontend** в профиле.

## Docker Compose не запускается

- Откройте Docker Desktop и дождитесь состояния Running.
- Проверьте `docker info` и `docker compose version`.
- У сервисов должны быть опубликованные host-порты, а не только `expose`.
- Если автоопределение выбрало не тот сервис, задайте имя сервиса или порт в полях **Frontend** и **Backend**.
- Используйте кнопку **Docker-логи** после попытки запуска.

Network Launcher создаёт временный Compose override для runtime URL. Он находится вне проекта и удаляется при корректной остановке.

## Ссылка создана, но сайт не становится готовым

Приложение считает сайт готовым при HTTP `2xx`, `3xx`, `401` или `403`. Постоянный `404` не считается готовностью.

Проверьте:

- правильный frontend-порт;
- что процесс не завершился после старта;
- что приложение слушает loopback/доступный host, а не другой интерфейс;
- ошибки сборки и загрузки зависимостей в журнале;
- готовность Docker-контейнеров.

## Страница открывается, но API или кнопки не работают

Чаще всего browser-side код обращается к `http://localhost:<порт>`. Для внешнего пользователя это его собственный компьютер. Используйте относительные пути:

```javascript
// проблемно для публичного посетителя
fetch('http://localhost:5000/api/products')

// проходит через GatewayProxy
fetch('/api/products')
```

Затем добавьте нужный префикс в **Backend-пути** и укажите backend-порт/сервис. Предупреждения `Preflight: browser URL указывает на loopback` показывают найденные файл и строку.

## WebSocket или HMR постоянно переподключается

- Оставьте **Dev/HMR = Авто** или включите его вручную.
- Проверьте, что путь WebSocket входит в backend-префиксы, если соединение обслуживает API.
- Для Socket.IO обычно нужен `/socket.io`.
- Попробуйте **Host upstream = Подменять на локальный**, если dev-сервер отклоняет публичный host.
- Проверьте origin/allowed-hosts настройки самого framework.

## Публичная ссылка возвращает страницу предупреждения ngrok

Network Launcher добавляет служебный заголовок при автоматической проверке, но политика бесплатного ngrok-плана может меняться. Откройте ссылку в обычном браузере и следуйте допустимому сценарию ngrok. Условия сервиса и лимиты задаются ngrok, а не Network Launcher.

## После остановки остались процессы или контейнеры

Сначала нажмите **Остановить** и дождитесь изменения статуса. Если приложение было аварийно завершено:

```powershell
Get-Process ngrok -ErrorAction SilentlyContinue
docker compose ps
```

Не завершайте чужие процессы и контейнеры. Перейдите в папку именно публиковавшегося Compose-проекта и остановите его штатной командой `docker compose down` только если уверены в цели.

## Какие данные приложить к issue

Перед созданием issue соберите:

- версию Network Launcher и Windows;
- тип проекта;
- последние относящиеся к ошибке строки вкладки **Логи** или `data/logs/app.log`;
- минимальную структуру проекта/Compose без секретов;
- шаги воспроизведения.

Удалите authtoken, приватные URL, пользовательские пути и данные проекта. Новый issue можно открыть [здесь](https://github.com/dartino21/network_launcher/issues/new).
